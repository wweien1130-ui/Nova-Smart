## 二、RabbitMQ 详解

### 2.1 消息流转完整流程

```
ServiceImpl ──调用──► Producer ──交换机+路由Key──► 交换机 ──路由Key匹配──► 队列 ──消费者取──► 执行逻辑
```

**详细流程（以购物车清理为例）：**

```
1️⃣ OrderServiceImpl（订单服务）
│
│ 创建订单后，调用
▼
orderMessageProducer.sendCartCleanMessage(
CartCleanMessage.builder()
.userId(userId)
.itemIds(List.of(1L, 2L, 3L))
.build()
);

2️⃣ OrderMessageProducer（生产者）
│
│ rabbitTemplate.convertAndSend(
│     "order.exchange",   ← 交换机
│     "cart.clean"        ← 路由Key
│ )
▼
消息 + 交换机名 + 路由Key 发给 RabbitMQ

3️⃣ order.exchange（交换机）
│
│ 收到消息，查看路由Key = "cart.clean"
│ 根据 Binding 绑定关系匹配
│
│ Binding: "cart.clean" ──► CART_CLEAN_QUEUE
▼
消息发送到 CART_CLEAN_QUEUE

4️⃣ CART_CLEAN_QUEUE（队列）
│
│ 存储消息，等待消费者
▼

5️⃣ OrderMessageConsumer（消费者）
│
│ @RabbitListener(queues = "cart.clean.queue")
│ 从队列取出消息
▼
cartService.removeByItemIds(itemIds, userId);  ← 真正执行业务
```

### 2.2 核心概念

| 组件 | 作用 | 类比 |
|------|------|------|
| **交换机 (Exchange)** | 消息分拣中心，根据规则路由消息 | 快递分拣中心 |
| **队列 (Queue)** | 存储消息的仓库，等待消费者取走 | 收货仓库 |
| **路由键 (Routing Key)** | 消息的"门牌号"，告诉交换机消息该去哪 | 快递单上的地址 |

### 2.3 生产者 vs 消费者

| 角色 | 需要关心什么 | 代码体现 |
|------|-------------|----------|
| **Producer** | 交换机 + 路由Key | `rabbitTemplate.convertAndSend(交换机, 路由Key, 消息)` |
| **Consumer** | 队列名 | `@RabbitListener(queues = "xxx.queue")` |

## 三、死信队列 (Dead Letter Queue)

### 3.1 什么是死信队列？

**死信队列 = "问题包裹专用通道"**

死信产生的原因：
- 消息被消费者拒绝（reject/nack）
- 消息超时（TTL 到期）
- 队列满了

### 3.2 本项目中的应用场景

**场景：订单超时未支付，自动取消**

```
订单创建 ──► ORDER_DELAY_QUEUE（延迟30分钟）
│
│ TTL 到期，无人处理
▼ 自动转发（死信）
ORDER_STATUS_QUEUE（死信队列）
│
▼
OrderMessageConsumer.handleOrderStatus()
│
▼
判断订单状态，如果是"待支付"则取消
```

### 3.3 配置死信队列

```java
@Bean
public Queue orderDelayQueue() {
    return QueueBuilder.durable(ORDER_DELAY_QUEUE)
            .withArgument("x-message-ttl", 30 * 60 * 1000)  // TTL = 30分钟
            .withArgument("x-dead-letter-exchange", ORDER_EXCHANGE)
            .withArgument("x-dead-letter-routing-key", ORDER_STATUS_KEY)
            .build();
}
```

## 四、Kafka 详解

### 4.1 消息流转完整流程

```
业务服务 ──调用──► Producer ──Topic+Key──► Topic ──分区──► Consumer
```

**详细流程（以聊天日志为例）：**

```
1️⃣ ChatController（聊天服务）
   │
   │ AI 回复后，调用
   ▼
   chatLogProducer.sendChatLog(ChatLogMessage.builder()
       .userId(userId)
       .sessionId(sessionId)
       .question(question)
       .answer(answer)
       .timestamp(timestamp)
       .build()
   );

2️⃣ ChatLogProducer（生产者）
   │
   │ kafkaTemplate.send(
   │     "chat-log",           ← Topic
   │     sessionId,             ← key（决定消息发到哪个分区）
   │     json消息
   │ )
   ▼

3️⃣ chat-log Topic
   │ Topic = 主题，类似 RabbitMQ 的交换机
   │ 根据 key 哈希决定发送到哪个分区
   ▼
   ┌─────────────┐
   │ Partition 0 │ ← 分区1
   ├─────────────┤
   │ Partition 1 │ ← 分区2
   ├─────────────┤
   │ Partition 2 │ ← 分区3
   └─────────────┘

4️⃣ ChatLogConsumer（消费者）
   │
   │ @KafkaListener(
   │     topics = "chat-log",
   │     groupId = "chat-log-group"
   │ )
   ▼
   log.info("收到聊天日志: {}", message);
```

### 4.2 核心概念对比

| Kafka 概念 | RabbitMQ 概念 | 说明 |
|-----------|--------------|------|
| **Topic** | Exchange | 消息的主题/分类 |
| **Partition** | Queue | 消息存储的分区 |
| **Key** | Routing Key | 决定消息发送到哪个分区 |
| **Consumer Group** | 消费者实例 | 消费组内负载均衡 |
| **Offset** | 无 | 消息的消费进度 |

### 4.3 分区机制

```java
@Bean
public NewTopic chatLogTopic() {
    return TopicBuilder.name(KafkaConstants.TOPIC_CHAT_LOG)
            .partitions(3)   // 3个分区
            .replicas(1)      // 1个副本
            .build();
}
```

**分区的意义：**
- **并行消费**：多个消费者可以分别处理不同分区
- **提高吞吐**：分区越多，并行处理能力越强
- **消息顺序**：同一分区内的消息有序，不同分区之间无序

**Key 的作用：** 相同 key 的消息会发送到同一分区
```java
kafkaTemplate.send("chat-log", sessionId, json);           // 同会话消息到同一分区
kafkaTemplate.send("user-action", userId.toString(), json); // 同用户消息到同一分区
```

## 五、RabbitMQ vs Kafka 对比

### 5.1 核心区别

| 特性 | RabbitMQ | Kafka |
|------|----------|-------|
| **设计目标** | 企业级可靠消息传递 | 高吞吐日志流处理 |
| **消息模型** | 交换机 + 队列 | Topic + 分区 |
| **消费模式** | 独占消费 | 消费组内负载均衡 |
| **消息持久化** | 可配置 | 默认持久化 |
| **吞吐量** | 万级/秒 | 百万级/秒 |
| **延迟消息** | 支持（TTL+死信） | 需外部插件 |

### 5.2 本项目选型

| 消息队列 | 用途 | 场景 |
|---------|------|------|
| **RabbitMQ** | 订单核心流程 | 库存扣减、购物车清理、订单超时检查 |
| **Kafka** | 日志记录、行为分析 | 聊天日志、用户行为、审计日志 |

### 5.3 选型原因

| 场景 | 选择 | 原因 |
|------|------|------|
| 订单库存扣减 | RabbitMQ | 需要可靠传递、失败重试、死信队列 |
| 购物车清理 | RabbitMQ | 必须成功执行，不能丢消息 |
| 聊天日志 | Kafka | 数据量大、允许少量丢失、追求高性能 |
| 用户行为 | Kafka | 高并发写入、分析场景不需要实时 |

---

## 六、总结

> **RabbitMQ = 交易型消息（必须可靠，不能丢消息）**
>
> **Kafka = 分析型消息（追求吞吐，允许少量丢失）**
>
> **路由 Key = 队列的"门牌号"，交换机根据门牌号把消息送到正确的队列**

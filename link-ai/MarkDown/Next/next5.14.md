## 一、项目微服务模块架构

### 1.1 模块组成

| 模块 | 名称 | 职责 |
|------|------|------|
| **hm-common** | 公共基础模块 | 共用 DTO/VO、异常处理、配置类、通用工具 |
| **hm-api** | 接口定义模块 | Feign 客户端接口定义，不含实现 |
| **hm-service** | 业务实现模块 | 提供真实业务逻辑（商品、购物车、订单、支付等） |
| **agent** | AI Agent 模块 | AI 对话代理服务，通过 Feign 调用 hm-service |

### 1.2 模块依赖关系

```
hm-common (公共层)
     │
     ├──→ hm-api (接口层)
     │         └──→ agent (通过 Feign 调用 hm-service)
     │
     ├──→ hm-service (业务层)
     │         └──→ agent (本地依赖)
     │
     └──→ agent (应用层)
               └──→ hm-service (Feign 远程调用)
```

### 1.3 调用关系详情

| 调用方 | 被调用方 | 调用方式 | 说明 |
|--------|---------|---------|------|
| agent | hm-api | 编译依赖 | 使用 Feign Client 接口定义 |
| agent | hm-common | 编译依赖 | 使用共用组件 |
| agent | hm-service | Feign 远程调用 | 运行时调用业务接口 |
| hm-api | hm-common | 编译依赖 | 定义接口参数和返回值 |
| hm-service | hm-common | 编译依赖 | 使用通用返回格式、异常处理 |

## 二、MQ 需求分析

### 2.1 当前项目 MQ 现状

| 依赖位置 | 状态 | 说明 |
|---------|------|------|
| hm-common/pom.xml | ✅ 已引入 | spring-amqp, spring-rabbit |
| hm-service/pom.xml | ❌ 已注释 | 未启用 |

### 2.2 适合 MQ 的业务场景

#### 场景一：订单创建流程

**当前实现**（同步串行）：
```java
1. 查询商品 → 2. 保存订单 → 3. 保存详情 → 4. 清理购物车 → 5. 扣减库存
```

**可优化为 MQ 异步处理**：
- 订单创建后，发送消息 → 异步扣减库存
- 发送消息 → 异步清理购物车
- 发送消息 → 异步发送通知

#### 场景二：支付流程

**当前实现**（同步调用）：
```java
扣减余额 → 修改支付单 → 修改订单状态
```

**可优化为 MQ 异步处理**：
- 支付成功，发送消息 → 异步更新订单状态
- 发送消息 → 异步发送支付通知
- 发送消息 → 异步更新物流状态

## 三、MQ 选型对比

### 3.1 RabbitMQ vs Kafka

| 维度 | RabbitMQ | Kafka |
|------|----------|-------|
| **消息可靠性** | ✅ 天然支持，确认机制完善 | ⚠️ 需额外配置 |
| **事务消息** | ✅ 支持 | ⚠️ 需用 Kafka Transactions |
| **延迟消息** | ✅ 支持（死信队列） | ❌ 不擅长 |
| **高吞吐量** | 一般（万级/秒） | ✅ 优秀（百万级/秒） |
| **消息堆积** | 适中 | ✅ 优秀（可长期存储） |
| **运维复杂度** | 简单 | 较复杂 |
| **学习成本** | 低 | 中等 |

### 3.2 纯 Kafka 方案的硬伤

```
❌ 问题1：订单支付场景
   Kafka 不支持原生延迟消息
   → 需要自己实现时间轮或外部调度系统
   
❌ 问题2：库存补偿场景
   Kafka at-least-once 导致库存可能重复扣减
   → 需要自己实现幂等补偿，复杂度高
   
❌ 问题3：超时取消订单
   需要额外的定时任务或外部系统
   → RabbitMQ 死信队列开箱即用
```

## 四、推荐方案：RabbitMQ + Kafka 混用

### 4.1 架构设计

```
┌─────────────────────────────────────────────────────────────────────┐
│                        推荐架构                                      │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   ┌─────────────────────────────────────────────────────────────┐   │
│   │                    RabbitMQ (核心业务)                        │   │
│   │                                                              │   │
│   │   • 订单创建 → 库存扣减 (可靠投递)                           │   │
│   │   • 支付成功 → 订单状态更新 (事务消息)                       │   │
│   │   • 订单取消 → 库存回滚 (补偿机制)                           │   │
│   │   • 购物车清理 (可靠消息)                                    │   │
│   │   • 超时未支付 → 自动取消订单 (延迟队列)                     │   │
│   │   • 库存不足 → 订单失败通知                                   │   │
│   │                                                              │   │
│   └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
│   ┌─────────────────────────────────────────────────────────────┐   │
│   │                      Kafka (数据平台)                         │   │
│   │                                                              │   │
│   │   • AI 对话日志采集 (高吞吐量)                               │   │
│   │   • 用户行为分析 (点击、浏览)                                │   │
│   │   • 业务数据变更日志 (审计)                                  │   │
│   │   • 实时数据统计                                             │   │
│   │                                                              │   │
│   └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 4.2 分工明细

#### RabbitMQ 适用场景

| 场景 | RabbitMQ 优势 |
|------|--------------|
| 库存扣减 | 消息确认机制确保库存不丢失 |
| 支付回调 | 事务消息保证支付和状态更新一致性 |
| 超时取消 | 死信队列实现延迟消息，开箱即用 |
| 失败重试 | 丰富的重试策略和死信处理 |
| 幂等处理 | 简单可靠的消息去重 |

#### Kafka 适用场景

| 场景 | Kafka 优势 |
|------|-----------|
| 对话日志 | 高吞吐量，百万级消息/秒 |
| 行为分析 | 实时流处理，Spark/Flink 生态 |
| 数据湖 | 长期存储，成本低 |

## 五、Spring Boot 3 集成配置示例

### 5.1 Maven 依赖

```xml
<!-- RabbitMQ -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-amqp</artifactId>
</dependency>

<!-- Kafka -->
<dependency>
    <groupId>org.springframework.kafka</groupId>
    <artifactId>spring-kafka</artifactId>
</dependency>
```

### 5.2 YAML 配置

```yaml
spring:
  rabbitmq:
    host: localhost
    port: 5672
    username: guest
    password: guest
  
  kafka:
    bootstrap-servers: localhost:9092
    consumer:
      group-id: hmall-group
      auto-offset-reset: earliest
    producer:
      acks: all
```

## 三、问题汇总

### 问题 1：Redis ChatMemoryRepository JSON 序列化失败

**现象**：调用 `RedisChatMemoryRepository.saveAll()` 时，使用 `JSONUtil.toJsonStr(message)` 序列化失败，存储到 Redis 的数据结构不完整。

**原因**：Spring AI 的 `Message` 接口实现类（如 `UserMessage`、`AssistantMessage`）继承层次复杂，直接使用 Hutool 的 `JSONUtil.toJsonStr()` 无法正确处理内部字段（如 `toolCalls`、`metadata`、`media` 等），反序列化时会丢失关键数据或报错。

**解决方案**：自定义中间实体 + 手动序列化/反序列化

1. 创建 `MyMessage` 实体，包含所有需要持久化的字段：
   ```java
   @Data
   public class MyMessage {
       private String messageType;
       private Map<String, Object> metadata;
       private List<Media> media;
       private List<AssistantMessage.ToolCall> toolCalls;
       private String textContent;
       private List<ToolResponseMessage.ToolResponse> toolResponses;
       private Map<String, Object> params;
   }


2. 创建 `MessageUtil` 工具类：

    - __`toJson(Message)`__：将 Message 通过 `BeanUtil.toBean()` 转为 `MyMessage`，再 `JSONUtil.toJsonStr()`
    - __`toMessage(String)`__：将 JSON 反序列化为 `MyMessage`，再根据 `messageType` 构建对应的 Message 子类

```java
// 序列化
public static String toJson(Message message) {
    var myMessage = BeanUtil.toBean(message, MyMessage.class);
    myMessage.setTextContent(message.getText());
    if (message instanceof AssistantMessage assistantMessage) {
        myMessage.setToolCalls(assistantMessage.getToolCalls());
    }
    if (message instanceof ToolResponseMessage toolResponseMessage) {
        myMessage.setToolResponses(toolResponseMessage.getResponses());
    }
    return JSONUtil.toJsonStr(myMessage);
}

// 反序列化
public static Message toMessage(String json) {
    var myMessage = JSONUtil.toBean(json, MyMessage.class);
    var messageType = MessageType.valueOf(myMessage.getMessageType());
    switch (messageType) {
        case USER -> {
            return UserMessage.builder()
                    .text(myMessage.getTextContent())
                    .metadata(myMessage.getMetadata())
                    .media(myMessage.getMedia())
                    .build();
        }
        case ASSISTANT -> {
            return new AssistantMessage(myMessage.getTextContent(), 
                myMessage.getMetadata(), myMessage.getToolCalls());
        }
        // ... 其他类型
    }
}
```

### 问题 2：Redis Sentinel 连接配置失败

__现象__：启动报错 `IllegalArgumentException: Redis sentinel nodes cannot be null or empty`。

__原因__：`application.yaml` 中的 Redis 配置项与 Spring Boot 自动配置的预期不匹配。`@Value("${spring.redis.sentinel.nodes}")` 读取不到值。

__修复过程__：

1. __第一轮尝试__：手动编写 `RedisConfig` 配置类，使用 `RedisSentinelConfiguration` 手动构建连接工厂

2. __遇到的问题__：`spring.redis.sentinel` 配置属性读取正常，但 `spring.data.redis.sentinel` 前缀的配置读取不到

3. __最终方案__：

    - 使用 `spring.redis.sentinel.*` 前缀（与 Spring Boot 自动配置兼容）
    - 手动创建 `LettuceConnectionFactory` 并注入到 `StringRedisTemplate`
    - 明确设置 key/value 的序列化器为 `StringRedisSerializer`

```yaml
spring:
  redis:
    sentinel:
      master: mymaster
      nodes: 192.168.115.128:27001,192.168.115.128:27002,192.168.115.128:27003
      password: 696903
    password: 696903
    timeout: 5000
    lettuce:
      pool:
        max-active: 8
        max-idle: 8
        min-idle: 0
```

### 问题 3：RedisConfigValidator 配置属性注入失败

__现象__：`RedisConfigValidator` 启动时打印日志显示所有配置为 null：

```javascript
Sentinel Master: 
Sentinel Nodes: 
```

__原因__：`RedisConfigValidator` 使用了 `@Value("${spring.data.redis.sentinel.master-name:}")` 前缀，但实际配置中的 key 是 `spring.redis.sentinel.master`。

__修复__：统一使用 `spring.redis.sentinel.*` 前缀，或者在 `@Value` 中指定备选 key。但该验证器主要用于调试，不影响核心功能。

### 问题 5：流式输出停止功能实现中的并发控制

__现象__：多次快速点击停止按钮，或同时发送多个请求时，生成状态混乱。

__原因__：`GENERATE_STATUS` 使用 `Map<String, Boolean>` 存储生成状态，需要确保线程安全。

__修复__：使用 `ConcurrentHashMap` 保证并发安全。

```java
private static final Map<String, Boolean> GENERATE_STATUS = new ConcurrentHashMap<>();

// 关键操作链：
.doFirst(() -> GENERATE_STATUS.put(sessionId, true))           // 开始生成
.doOnError(throwable -> GENERATE_STATUS.remove(sessionId))     // 异常清理
.doOnCancel(() -> saveStopHistoryRecord(conversationId, ...))  // 取消时保存
.doOnComplete(() -> GENERATE_STATUS.remove(sessionId))         // 完成清理
.takeWhile(response -> GENERATE_STATUS.getOrDefault(sessionId, false)) // 控制流
```

__流程说明__：

1. 用户点击发送 → `GENERATE_STATUS.put(sessionId, true)`
2. AI 流式输出 → `takeWhile` 检查状态，false 时停止
3. 用户点击停止 → 调用 `stop(sessionId)` → `GENERATE_STATUS.remove(sessionId)`
4. 流被中断 → `doOnCancel` 回调 → 将已输出的内容存入 Redis
5. 同时前端 AbortController 中止 fetch 请求

### 问题 6：前端停止按钮和发送按钮切换

__现象__：生成时发送按钮仍然可见，用户可能误操作。

__修复__：生成流式输出时显示停止按钮，隐藏发送按钮；流结束后恢复。

```javascript

stopBtn.style.display = 'flex';

// 生成结束（finally块）
sendBtn.style.display = 'flex';
stopBtn.style.display = 'none';
```

### 问题 7：conversationId 生成策略的统一

__现象__：之前 `ChatServiceImpl` 中 `systemPrompt` 使用了 `sessionId` 作为参数。

__原因__：需要区分 `sessionId`（前端会话标识，对应数据库中的 `ChatSession`）和 `conversationId`（Spring AI ChatMemory 使用的会话标识）。

__修复__：新增 `ChatService.getConversationId(sessionId)` 静态方法，统一转换规则：

```java
// ChatService 接口中
static String getConversationId(String sessionId) {
    return sessionId;
}

// 使用示例
String conversationId = ChatService.getConversationId(sessionId);
```

## 四、Redis 聊天记忆架构

### 数据流向

```javascript
用户 → ChatController → ChatServiceImpl
                              │
                              ▼
                    ChatClient (DeepSeek-V4)
                              │
                              ▼
                    Flux<ChatEventVo> (SSE)
                              │
                              ▼
               RedisChatMemoryRepository
                              │
                              ▼
                    Redis Sentinel Cluster
                     192.168.115.128
                       27001/27002/27003
```

### Redis 存储结构

```javascript
CHAT:{conversationId} → List
  ┌─────────────────────────────┐
  │ {"messageType":"USER",...}  │ ← 用户消息
  │ {"messageType":"ASSISTANT", │ ← AI 回复
  │  "textContent":"...",       │
  │  "toolCalls":[...]}         │
  │ {"messageType":"USER",...}  │ ← 下一轮用户消息
  │ ...                         │
  └─────────────────────────────┘
```

### 缓存策略

- __最大消息数__：`tj.ai.memory.max=100`（通过 `MessageWindowChatMemory` 自动管理）
- __超出限制__：自动删除最旧的对话
- __中断保存__：用户停止生成时，已输出的内容通过 `doOnCancel` 保存到 Redis

## 六、经验总结

### 6.1 Redis 序列化

1. __Spring AI Message 序列化复杂__：直接使用 `JSONUtil.toJsonStr()` 不可行，需要自定义中间实体
2. __手动序列化/反序列化__：使用自定义 `MyMessage` + `MessageUtil`，针对每种 `MessageType` 分别处理
3. __Redis Sentinel 连接__：Spring Boot 2.x/3.x 的配置前缀变化，建议手动配置 `LettuceConnectionFactory`

### 6.2 配置管理

1. __基础设施 vs 业务配置分离__：datasource/redis 放在本地 `application.yaml`，AI 业务配置放在 Nacos
2. __Nacos 引用顺序__：多个 `optional:nacos:*` 按优先级排列，避免配置覆盖
3. __环境配置__：使用 `application-dev.yaml` / `application-local.yaml` 区分不同环境

### 6.3 流式输出控制

1. __ConcurrentHashMap 保证并发安全__：多线程环境下状态管理必须使用线程安全容器
2. __takeWhile + GENERATE_STATUS__：通过控制 Flux 流来实现输出停止，优雅且高效
3. __doOnCancel 保存中间结果__：用户中断时保留已生成的内容，不丢失数据
4. __双通道停止__：前端 AbortController + 后端 GENERATE_STATUS，确保停止立即可达
5. __前端按钮切换__：发送/停止按钮互斥，避免 UI 层面的误操作

### 6.4 代码组织

1. __工具类与实体分离__：`MyMessage`（数据载体）和 `MessageUtil`（转换逻辑）职责清晰
2. __枚举统一管理__：`MessageTypeEnum` 统一消息类型，避免硬编码
3. __VO 封装__：`MessageVO` 对外隐藏内部实现细节，只暴露前端需要的数据

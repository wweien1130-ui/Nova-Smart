## 1. ES 搜索不准：搜"衣服"出"老花镜"，搜"华为"出"小米"

### 问题描述

搜索"衣服"时，ES 返回了"老花镜"；搜索"华为"时，返回了"小米手机"。搜索结果完全不相关。

### 根因分析

ES 搜索使用的是向量相似度搜索（embedding + cosine similarity），没有做**业务元数据过滤**。

- ES 索引中虽然有 `category`（分类）、`brand`（品牌）字段
- 但搜索时没有利用这些字段做 filter 过滤
- 纯向量搜索只关注语义相似度，不关心业务分类
- 例如"衣服"的向量可能和"老花镜"的向量在某些维度上相似，导致误匹配

### 解决方案

**元数据过滤（优先级最高）**

原理：为每条商品数据添加 `category`、`brand` 等业务标签，搜索时先限定范围。

#### 1.1 ES 映射中添加过滤字段

`ItemDocument.java` 中已有 `category`、`brand` 字段，ES 映射中已存在。

#### 1.2 搜索时增加 filter 条件

在 `VectorSearchService.java` 中新增带过滤的重载方法：

```java
public SearchResponse<ItemDocument> searchByText(String text, int topK, String category, String brand) {
    // 构建 filter 条件
    List<Query> filters = buildFilter(category, brand);
    
    // 使用 filter 进行预过滤，再执行向量搜索
    return elasticsearchTemplate.search(
        NativeQuery.builder()
            .withQuery(q -> q.bool(b -> {
                b.must(m -> m.match(t -> t.field("name").query(text)));
                if (!filters.isEmpty()) {
                    b.filter(filters);
                }
                return b;
            }))
            .build(),
        ItemDocument.class
    );
}
```

#### 1.3 缓存机制

为了避免每次搜索都全量扫描 ES，实现了缓存机制：

- **懒加载**：首次搜索时加载所有数据到内存
- **定时刷新**：每 5 分钟自动刷新缓存
- **关键词匹配**：在缓存中做毫秒级的关键词匹配

```java
@Component
public class ItemCache {
    private final Map<String, List<ItemDocument>> cache = new ConcurrentHashMap<>();
    
    @PostConstruct
    public void init() {
        loadAllItems();  // 首次加载
    }
    
    @Scheduled(fixedRate = 300000)  // 5分钟刷新
    public void refresh() {
        loadAllItems();
    }
}
```

#### 1.4 双层策略

1. **关键词匹配（毫秒级）**：先在缓存中做关键词匹配
2. **LLM 兜底**：关键词匹配不到时，再调 LLM 做语义理解

#### 1.5 测试 API

新增 4 个测试端点：

| 端点 | 说明 |
|------|------|
| `GET /rag/search?q=衣服` | 纯 ES match 分词查询 |
| `GET /rag/search/filter?q=衣服&category=服装` | 带分类过滤的 ES 查询 |
| `GET /rag/search/brand?q=华为&brand=华为` | 带品牌过滤的 ES 查询 |
| `GET /rag/search/smart?q=衣服` | 智能搜索（关键词匹配 + LLM 兜底） |

## 2. 购物车数据不一致：加购后查看购物车为空

### 问题描述

用户说"我要这个"（加购），大模型回复"已加入购物车"。但用户说"查看购物车"时，大模型说"购物车是空的"。

### 根因分析

**大模型根本没调用 `addToCart` tool，而是自己编造了"已加入购物车"的回复。**

大模型的工作流程：
1. 用户说"我要这个" → 大模型应该调用 `addToCart` tool
2. 但大模型**直接回复"好的，已加入购物车"**，根本没调 tool
3. 用户说"查看购物车" → 大模型调用 `viewCart` → 返回空列表
4. 但大模型**可能不展示真实结果，而是根据之前的对话编造内容**

### 解决方案

在 CART 智能体的系统提示词中增加**强制规则**：

```java
.defaultSystem("""
    你是购物车管理专家，负责帮助用户管理购物车。

    【重要规则：必须调用工具，不能自己编造】
    1. 所有购物车操作（加购、查看、移除）都必须通过调用对应的工具完成
    2. 绝对不能自己编造"已加入购物车"、"购物车有XX商品"等回复
    3. 工具调用成功后才能回复用户操作成功
    4. 如果工具返回空结果，必须如实告知用户"购物车是空的"

    可用的工具：
    1. addToCart(itemId, name, price, image, spec, num) - 添加商品到购物车
       - 所有参数必须提供，不能省略
       - 如果用户没有提供完整参数，可以从对话历史中获取
       - 如果用户说"可以"、"好的"、"加购"等确认语，直接调用 addToCart 工具
       - 不要询问用户"是否要加购"，直接执行加购操作
       - price 单位是分，注意转换

    2. viewCart() - 查看购物车内容
       - 调用后必须如实展示工具返回的结果
       - 如果返回空列表，必须告诉用户"购物车是空的"

    3. removeFromCart(itemId) - 从购物车移除商品
       - 先调用 viewCart() 获取购物车列表，从返回结果中获取要移除商品的 itemId
       - 然后用获取到的 itemId 调用 removeFromCart
       - 如果用户说"移除XX商品"、"删除XX"等，直接执行移除操作，不要询问确认
    """)
```

**修改文件**：`hmall/agent/src/main/java/com/liang/agent/config/RouterAgentConfig.java`

## 3. 地址查询问题：数据库有地址但 agent 说没有

### 问题描述

用户说"默认地址"，大模型回复"目前还没有添加任何收货地址"。但数据库中 `address` 表明明有该用户的默认地址。

### 根因分析

**和购物车问题一样——大模型根本没调用 `listAddresses` tool，而是自己编造了"没有地址"的回复。**

ADDRESS 智能体的系统提示词之前只有一句话：

```java
"你是地址管理专家，负责帮助用户管理收货地址，包括查询地址、添加地址、更新地址、删除地址等操作。"
```

大模型觉得"我知道用户没有地址"，就直接回复了，根本没调 tool 去查数据库。

### 解决方案

给 ADDRESS 智能体也加上**强制调用 tool 的规则**：

```java
.defaultSystem("""
    你是地址管理专家，负责帮助用户管理收货地址。

    【重要规则：必须调用工具，不能自己编造】
    1. 所有地址操作（查询、添加、更新、删除）都必须通过调用对应的工具完成
    2. 绝对不能自己编造"没有地址"、"地址已更新"等回复
    3. 工具调用成功后才能回复用户操作成功
    4. 如果工具返回空列表，必须如实告知用户"没有找到地址"

    可用的工具：
    1. listAddresses() - 查询当前用户的所有收货地址列表
       - 调用后必须如实展示工具返回的结果
       - 如果返回空列表，必须告诉用户"没有找到地址"

    2. addAddress(contact, mobile, province, city, town, street, isDefault) - 添加新的收货地址

    3. updateAddress(addressId, contact, mobile, province, city, town, street, isDefault) - 更新收货地址

    4. deleteAddress(addressId) - 删除收货地址
    """)
```

**修改文件**：`hmall/agent/src/main/java/com/liang/agent/config/RouterAgentConfig.java`

## 4. 模型配置问题：Nacos 配置不生效，代码硬编码

### 问题描述

Nacos 配置中写的是 `deepseek-v4-flash`，但实际使用的是 `deepseek-v4-pro`。修改 Nacos 配置后，模型没有切换。

### 根因分析

`AIServiceConfig.java` 中模型名是**硬编码**的：

```java
// DashScope 模型
DashScopeChatOptions.builder()
    .withModel("deepseek-v4-pro")           // ← 写死了
    .build()

// OpenAI 模型
OpenAiChatOptions.builder()
    .model("qwen3.6-flash-2026-04-16")      // ← 写死了
    .build()
```

Nacos 配置 `hm-agent.yaml` 中的 `tj.ai.models.dashscope` 和 `tj.ai.models.openai` 根本没有被代码读取。

### 解决方案

改为从 Nacos 配置动态读取：

```java
@Value("${tj.ai.models.dashscope}")
private String dashscopeModel;

@Value("${tj.ai.models.openai}")
private String openaiModel;

// DashScope 模型
DashScopeChatOptions.builder()
    .withModel(dashscopeModel)              // ← 从 Nacos 读取
    .build()

// OpenAI 模型
OpenAiChatOptions.builder()
    .model(openaiModel)                     // ← 从 Nacos 读取
    .build()
```

**修改文件**：`hmall/agent/src/main/java/com/liang/agent/config/AIServiceConfig.java`

**效果**：以后只需要修改 Nacos 中 `hm-agent.yaml` 的这两行，重启应用即可切换模型：

```yaml
tj:
  ai:
    models:
      dashscope: "deepseek-v4-flash"    # 改这里就行
      openai: "qwen3.5-122b-a10b"       # 改这里就行
```

## 总结

| 问题 | 根因 | 解决方案 | 涉及文件 |
|------|------|---------|---------|
| ES 搜索不准 | 没有业务元数据过滤 | 增加 category/brand filter + 缓存 + 双层策略 | `VectorSearchService.java`, `RagTestController.java` |
| 购物车数据不一致 | 大模型自己编造，没调 tool | 系统提示词强制要求调用 tool | `RouterAgentConfig.java` |
| 地址查询为空 | 大模型自己编造，没调 tool | 系统提示词强制要求调用 tool | `RouterAgentConfig.java` |
| Nacos 配置不生效 | 代码硬编码模型名 | 改为 `@Value` 从 Nacos 读取 | `AIServiceConfig.java` |

### 核心教训

**大模型会自己编造答案，不会主动调用 tool。** 必须在系统提示词中明确强制要求：

1. 所有数据操作都必须通过调用 tool 完成
2. 绝对不能自己编造数据
3. 工具返回空结果时必须如实告知用户

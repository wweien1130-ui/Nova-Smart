# 从单一智能体扩展为多智能体（Router Agent）改造总结

## 一、改造背景


用户输入后的完整调用链路（含代码文件）
## 用户输入后的完整调用链路（含代码文件）

```
用户发送 POST /agent/chat 请求（含 message, sessionId, userId）
    ↓
AgentController.java  ← 控制器，接收请求
    ↓ 调用 routerAgentService.chat(message, sessionId, userId)
RouterAgentService.java  ← 路由服务，核心调度
    ↓
    ├── 1. chatSessionService.update(sessionId, question, userId)
    │      ↓
    │   ChatSessionServiceImpl.java  ← 更新会话标题（同步）
    │
    ├── 2. routeAndProcess(question, sessionId, userId, conversationId)
    │      ↓
    │   ├── 2.1 routerChatClient.prompt().user(question).call().content()
    │   │      ↓
    │   │   RouterAgentConfig.java  ← 路由智能体配置（@Bean("routerChatClient")）
    │   │      ↓ 调用大模型（qwen3.6-flash）
    │   │      返回 JSON: {"agentType":"CART","question":"查询购物车"}
    │   │
    │   ├── 2.2 parseRouterResult(json) → AgentTypeEnum
    │   │      ↓
    │   │   AgentTypeEnum.java  ← 智能体类型枚举
    │   │
    │   ├── 2.3 parseRouterQuestion(json) → String
    │   │
    │   ├── 2.4 getTargetClient(agentType) → ChatClient
    │   │      ↓ switch 分发
    │   │   ├── ITEM    → itemChatClient    (ItemSearchTool)
    │   │   ├── CART    → cartChatClient    (CartTool)
    │   │   ├── ORDER   → orderChatClient   (OrderTool)
    │   │   └── ADDRESS → addressChatClient (AddressTool)
    │   │      ↓
    │   │   RouterAgentConfig.java  ← 各子智能体配置
    │   │
    │   └── 2.5 targetClient.prompt().user(question).stream().chatResponse()
    │          ↓ 调用大模型（deepseek-v4-pro）
    │          大模型决定是否调用 Tool
    │          ↓
    │       ├── CartTool.java       ← 购物车 Tool（addToCart/viewCart/removeFromCart）
    │       ├── ItemSearchTool.java ← 商品搜索 Tool（searchItems/getItemDetail）
    │       ├── OrderTool.java      ← 订单 Tool（createOrder/queryOrder/cancelOrder...）
    │       └── AddressTool.java    ← 地址 Tool（queryAddress/addAddress/updateAddress...）
    │
    └── 3. Flux 流式输出给前端
           ↓
        ChatEventVo.java  ← 事件 VO（eventType/eventData）
```

| 序号 | 文件路径 | 作用 |
|------|---------|------|

| 2 | `AgentRequest.java` | 请求 DTO（message/sessionId/userId） |

| 5 | `AgentTypeEnum.java` | 智能体类型枚举 |

| 9 | `CartTool.java` | 购物车 Tool（addToCart/viewCart/removeFromCart） |
| 10 | `ItemSearchTool.java` | 商品搜索 Tool（searchItems/getItemDetail） |

| 13 | `ChatEventVo.java` | 流式输出事件 VO |









原先的架构是**单一智能体 + 多 Tool** 模式：

```
用户提问 → ChatClient（一个模型，一个 System Prompt）
              ↓
        14 个 Tool 全部注册（ItemSearchTool, CartTool, AddressTool, OrderTool）
              ↓
        大模型自行决定调用哪个 Tool
```

**问题：**
- 14 个 Tool 混在一起，大模型容易选错
- 所有功能共享同一个 System Prompt，无法精细化控制
- 新增业务领域时，Tool 数量继续膨胀，选错概率上升

---

## 二、改造后的架构

```
用户提问 → Router Agent（路由智能体，不注册任何 Tool）
              ↓ 分析意图，返回 JSON
    ┌───────┼───────┐
    ↓       ↓       ↓       ↓
  ITEM    CART    ORDER   ADDRESS
  (2Tool) (3Tool) (5Tool) (4Tool)
```

**每次对话调用 2 次大模型：**
1. **Router Agent**（非流式）→ 分析意图，返回 `{"agentType":"ORDER","question":"..."}`
2. **子 Agent**（流式）→ 执行业务逻辑，流式输出回答

## 三、改动清单

### 1. 新增 `AgentTypeEnum.java`
定义智能体类型枚举，用于 Router 的返回值和路由决策。

```java
public enum AgentTypeEnum {
    ROUTE("ROUTE", "路由智能体"),
    ITEM("ITEM", "商品搜索智能体"),
    CART("CART", "购物车管理智能体"),
    ORDER("ORDER", "订单管理智能体"),
    ADDRESS("ADDRESS", "地址管理智能体");
}
```

### 2. 新增 `RouterAgentConfig.java`
配置 5 个独立的 ChatClient，每个子 Agent 只注册自己的 Tool：

| Bean 名称 | 注册的 Tool | Tool 数量 |
|-----------|------------|----------|
| routerChatClient | 无 | 0 |
| itemChatClient | ItemSearchTool | 2 |
| cartChatClient | CartTool | 3 |
| orderChatClient | OrderTool | 5 |
| addressChatClient | AddressTool | 4 |

**关键代码：**
```java
@Bean("routerChatClient")
public ChatClient routerChatClient(ChatModel openapiChatModel) {
    return ChatClient.builder(openapiChatModel)
            .defaultSystem("分析用户意图，返回 JSON...")
            .build();  // 不注册任何 Tool
}

@Bean("cartChatClient")
public ChatClient cartChatClient(ChatModel dashScopeChatModel, ..., CartTool cartTool) {
    return ChatClient.builder(dashScopeChatModel)
            .defaultSystem("你是购物车管理专家...")
            .defaultTools(cartTool)  // 只注册 CartTool
            .build();
}
```

### 3. 新增 `RouterAgentService.java`
路由核心逻辑，实现"Router 分析意图 → 路由到子 Agent → 流式输出"的完整流程。

**核心机制：**
```java
// 1. Router 分析意图（非流式）
String routerResult = routerChatClient.prompt()
        .user(question).call().content();
// → 返回 JSON: {"agentType": "CART", "question": "查询购物车"}

// 2. 解析 JSON
AgentTypeEnum agentType = parseRouterResult(routerResult);

// 3. switch 分发到对应的子 Agent
ChatClient targetClient = getTargetClient(agentType);
// → case CART -> cartChatClient;

// 4. 子 Agent 流式处理
targetClient.prompt()
        .user(actualQuestion)
        .toolContext(userId)
        .stream().chatResponse()
        ...
```

### 4. 简化 `ChatServiceImpl.java`
从 247 行简化为 20 行，直接委托给 `RouterAgentService`：

```java
@Service
public class ChatServiceImpl implements ChatService {
    private final RouterAgentService routerAgentService;

    public Flux<ChatEventVo> chat(String question, String sessionId, Long userId) {
        return routerAgentService.chat(question, sessionId, userId);
    }
}
```

### 5. 清理 `AIServiceConfig.java`
移除旧的 `chatClient()` Bean（注册了所有 14 个 Tool 的方法），因为已经分散到 `RouterAgentConfig` 中。

### 6. 修改 `AgentController.java`
将注入的 `ChatClient` 改为 `RouterAgentService`，解决 5 个 ChatClient Bean 的冲突问题。

### 7. 新增 `AgentRequest.userId` 字段
支持前端传入用户 ID。

## 四、运行中遇到的问题及解决

### 问题 1：Router 返回的 JSON 被 markdown 代码块包裹
**现象：** 大模型返回 ` ```json\n{"agentType":"ITEM","question":"你好"}\n``` `，`JSONUtil.parseObj()` 无法解析。

**解决：** 新增 `extractJsonFromResponse()` 方法，自动提取代码块内的纯 JSON：
```java
private String extractJsonFromResponse(String text) {
    // 支持三种格式：
    // 1. ```json\n{...}\n```  — 带 json 标记的代码块
    // 2. ```\n{...}\n```      — 不带标记的代码块
    // 3. {...}                — 纯 JSON 文本
}
```

### 问题 2：子 Agent 调用时报 `text cannot be null or empty`
**现象：** 路由决策成功后，子 Agent 调用时报参数为空异常。

**解决：** `getSystemPromptForAgent()` 对非 ITEM 类型返回空字符串 `""`，`.text("")` 报错。改为返回 `null`，并在调用处判断：
```java
if (systemText != null && !systemText.isBlank()) {
    promptBuilder.system(...);  // 有自定义提示词才设置
}
// 否则使用 ChatClient 的 defaultSystem
```

### 问题 3：添加商品到购物车时大模型犹豫不决
**现象：** 大模型知道要调用 `addToCart`，但觉得参数太多不敢直接调用，反而问用户"是否要加购"。

**解决：** 优化 Cart Agent 的 System Prompt：
- 明确说明"所有参数必须提供，可以从对话历史中获取"
- 规定"用户说'可以'、'好的'、'加购'时直接调用，不要犹豫"
- 提示 price 单位是分，注意转换

### 问题 4：移除商品时大模型不知道 itemId 从哪里获取
**现象：** 大模型说"已移除"，但实际上没有调用 `removeFromCart` Tool。

**解决：** 优化 Tool 描述和 System Prompt：
- `@Tool` 描述改为 `"从购物车移除商品，需要先调用 viewCart 查看购物车获取商品ID"`
- `@ToolParam` 描述改为 `"要移除的商品ID，从 viewCart 返回结果中获取 itemId"`
- System Prompt 规定"先调用 viewCart() 获取购物车列表，再调用 removeFromCart"

## 五、关键设计决策

| 决策点 | 方案 | 原因 |
|-------|------|------|
| **Router 调用方式** | 非流式 `call().content()` | Router 只需要返回 JSON，不需要流式 |
| **子 Agent 调用方式** | 流式 `stream().chatResponse()` | 需要流式输出给前端 |
| **Router 返回格式** | JSON `{"agentType":"ITEM","question":"..."}` | 结构化，易于解析 |
| **对话历史** | 所有子 Agent 共享同一个 `conversationId` | 保证对话连续性 |
| **userId 传递** | 通过 `toolContext` 传递 | 与现有 Tool 的 userId 获取方式一致 |
| **兜底策略** | Router 解析失败时默认路由到 ITEM | 不影响用户体验 |
| **Router 模型** | qwen3.6-flash（轻量快速） | 只做意图分类，不需要强推理能力 |
| **子 Agent 模型** | deepseek-v4-pro | 业务逻辑需要高准确性 |

## 六、后续优化方向

| 方向 | 说明 | 优先级 |
|------|------|--------|
| **System Prompt 精细化** | 为每个子 Agent 定制更详细的提示词 | 高 |
| **Tool 描述优化** | 让大模型更清楚每个 Tool 的用途和数据来源 | 高 |
| **参数校验增强** | 在 Tool 方法内部做防御性校验 | 中 |
| **Few-shot 示例** | 在 System Prompt 中给几个调用示例 | 中 |
| **输出格式约束** | 要求大模型先思考再调用（ReAct 模式） | 低 |

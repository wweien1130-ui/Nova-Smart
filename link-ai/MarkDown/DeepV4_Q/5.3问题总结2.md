# 5.3 问题总结2

## 1. 编译错误：ChatServiceImpl.java 中变量引用错误

### 问题描述
`.map(chatResponse -> { ... })` 中使用了 `response` 变量，但 lambda 参数名是 `chatResponse`，导致编译错误。

### 修复
将 `response.getMetadata().getId()` 改为 `chatResponse.getResult().getOutput().getMetadata().get(Constant.ID)`。

### 原因
- `ChatResponse.getMetadata()` 返回 `ChatResponseMetadata`，没有 `getId()` 方法
- 需要从 `AssistantMessage.getMetadata()`（返回 `Map<String, Object>`）中通过 key `"id"` 获取消息 ID

---

## 2. params 存储到 Redis 的完整链路

### 问题描述
Tool 方法执行时产生的参数数据（如商品信息）需要随历史消息一起保存到 Redis，以便在历史聊天记录中恢复时展示操作详情。

### 数据流
```
流结束 (finishReason = "STOP")
  → .map() 中获取 messageId
  → ToolResultHolder.put(messageId, "requestId", requestId)  // 建立 messageId → requestId 映射
  → .concatWith() 中 ToolResultHolder.get(requestId) 获取 params
  → 发送 params 到前端
  → MessageUtil.toJson() 保存到 Redis 时：
      → 通过 messageId 获取 requestId
      → 通过 requestId 获取 params
      → 将 params 设置到 MyAssistantMessage 中
      → 保存到 Redis
```

### 问题：时序冲突导致 params 为 null
`.concatWith()` 中先 `remove(requestId)` 把 params 取走发给了前端，后续 `MessageUtil.toJson()` 再通过 `requestId` 去获取时，数据已经不存在了。

### 修复方案（方案A）
- **ChatServiceImpl.java**：`.concatWith()` 中不移除 `requestId` 对应的数据
- **MessageUtil.java**：读取 params 后统一清理 `messageId` 和 `requestId`

## 3. IllegalArgumentException: key is not null!

### 问题描述
`ToolResultHolder.remove(key)` 中使用了 `Assert.notNull(key, "key is not null!")`，当 key 为 `null` 时抛出异常。

### 原因
- `assistantMessage.getMetadata().get(Constant.ID)` 可能返回 `null`
- `Convert.toStr(null)` 返回 `null`
- `ToolResultHolder.get(messageId, Constant.REQUEST_ID)` 可能返回 `null`

### 修复
在调用 `remove` 前判空：
```java
var messageIdObj = assistantMessage.getMetadata().get(Constant.ID);
if (messageIdObj != null) {
    var messageId = Convert.toStr(messageIdObj);
    var requestId = Convert.toStr(ToolResultHolder.get(messageId, Constant.REQUEST_ID));
    // ... 读取 params ...
    ToolResultHolder.remove(messageId);
    if (requestId != null) {
        ToolResultHolder.remove(requestId);
    }
}
```

## 4. 历史消息重复加载问题

### 问题描述
重启后端后再次询问，AI 把前面的回答和后面的回答融合在一起输出。

### 原因
`getConversationId` 依赖 `UserContext.getUser()`，但在 Agent 调用链路中不走 `LoginInterceptor`，所以 `UserContext.getUser()` 返回 `null`，导致 `conversationId = "null_sessionId"`。

每次请求都生成相同的 `"null_sessionId"`，AI 的上下文中包含了所有历史消息，导致回答内容合并。

### 修复
1. **ChatService.java** — `getConversationId` 改为接收 `userId` 参数，不再依赖 `UserContext`
   ```java
   static String getConversationId(String sessionId, Long userId) {
       return (userId != null ? userId : 0) + "_" + sessionId;
   }
   ```
2. **ChatServiceImpl.java** — 调用时传入 `userId`
3. **ChatSessionServiceImpl.java** — 调用时传入 `UserContext.getUser()`
